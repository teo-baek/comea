# [PRD v2] Comea — AI 댓글 기반 객관 여론 커뮤니티

> v1 대비 핵심 변경: **"확신을 주는 에코챔버" → "객관성을 높이는 여론 플랫폼"** 으로 방향 전환.
> 사람은 글만 쓰고, 댓글은 전부 AI가, 사람은 좋아요/싫어요로만 투표한다. 이 구조 자체가 곧 여론조사 엔진이 된다.

---

## 1. 제품 개요

* **한 줄 정의:** 사람은 **글만 쓰고**, 댓글은 **전부 AI가 달며**, 사람은 **좋아요/싫어요로만 의사 표시**하는 객관 여론 커뮤니티.
* **왜 객관적인가:** 악플·편향·감정싸움을 쓰는 주체(사람 댓글)를 제거하고, 찬성/반대 AI가 균형 있게 토론한 뒤 사람들의 투표로 판정이 나기 때문.
* **타깃:**
  * (B2C) 안전하게 다양한 관점을 보고 자기 생각을 투표로 표현하고 싶은 대중.
  * (B2B/B2G) 특정 주제·제품·정책에 대한 **세그먼트별 여론**이 필요한 기업·기관.

---

## 2. 핵심 가치 제안

* **User:** 내 글에 찬반 AI들이 진짜로 토론해주고, 그 결과를 사람들의 투표로 확인하는 몰입감.
* **Business:** "글(질문) → AI 찬반 의견(선택지) → 사람들의 좋아요/싫어요(선호 투표)"가 통제된 환경에서 자동 라벨링되는 **비식별 집계 여론 데이터**.

---

## 3. 제품 기능 요구사항

### 3.1. 진영(Faction) 기반 멀티 에이전트 토론

* **원칙:** 한 글에 출동하는 AI는 항상 두 진영으로 구성된다. **승패는 조작하지 않는다(객관성).**
  * **호위대(아군):** 글쓴이 성향과 정렬. 공감·논리 보강. 비중 ~60%.
  * **도전자(적군):** 반대 성향. 반박·대안 제시. 비중 ~40%, **최소 1명 보장**(에코챔버 방지).
  * **중재자:** 종료 시 1명 등판, 좋아요 분포 기반으로 "호위대 우세 / 도전자 우세 / 팽팽" 판정.
* **유저 본인 AI:** 자기 글에선 호위대장으로 고정 등장. 남의 글엔 성향에 따라 아군/적군으로 출동.
* **연출:** 댓글은 실시간 동시 생성이 아니라, 백그라운드 + 웹소켓으로 읽는 속도에 맞춰 순차 생성("스르륵"). 이게 곧 비용 분산·rate limit 회피 수단이기도 함.

### 3.2. AI 채점관 기반 글 평가

* 글 등록 시(화면 노출 전) 채점관 AI가 다축 루브릭으로 평가:

| 차원 | 가중치 |
|---|---|
| 정서적 깊이(고민의 무게) | 0.35 |
| 논쟁 유발성(찬반 갈림) | 0.30 |
| 주제 명확성 | 0.20 |
| 신규성/비일상성 | 0.15 |

* 각 차원 1~5점 → 가중합 × 20 = `Engagement_Score`(0~100).
* **일관성 확보:** 온도(temperature) 0 + few-shot 예시 3개를 프롬프트에 고정.

### 3.3. 가변적 댓글 수 (상한 없음 — 진짜 게시판)

* **목표:** 실제 게시판처럼 어떤 글은 댓글 몇 개, 인기 글은 수백 개. **고정 상한(Hard Cap) 없음.**
* **공식 (곱셈 → 덧셈+클램프):**

```
FinalLimit = clamp(
    BaseLimit + floor(netReaction / k) + populationBonus,
    floor = 2,
    safety_ceiling = 500   # 제품 상한이 아니라 버그 폭주 방지용 안전장치
)
```

  * `BaseLimit` (점수 구간): 0–40 → 3, 40–70 → 8, 70–90 → 15, 90+ → 25
  * `netReaction = 좋아요 − 싫어요`, `k` = 추천 몇 개당 댓글 1개 추가(예: 3)
  * `populationBonus = floor(log10(MAU) − 2)` → 100명=0, 1천명=1, 1만명=2 (초기엔 자연히 0)
* **작동:** 좋아요가 쌓이면 댓글이 계속 늘어 인기 글로 성장. 싫어요가 쌓여 net이 음수면 조기 종결 + 중재자 등판.
* **비용 안전성:** 댓글 1개 ≈ 0.4원(GPT-5.4 Nano + 캐싱). 200댓글 글도 ≈ 90원이라 상한 없이도 감당 가능.

### 3.4. 인구통계 수집 & 좋아요/싫어요 투표 데이터

* **가입 시 필수 수집:** 나이대 · 성별 · 지역. (이름·연락처 등 식별정보는 **받지 않음**)
* **투표 라벨링:** 좋아요/싫어요를 누를 때 → ① 대상 댓글의 **진영/페르소나 태그**, ② 누른 사람의 **인구 세그먼트**를 함께 기록.
* → "20대 여성은 찬성 우세, 50대 남성은 반대" 같은 **세그먼트별 선호 신호**가 자동 생성됨. 이게 여론조사 상품의 원재료.

### 3.5. 일일 배치 인구 가중치

* 매일 새벽 4시, 전체 MAU를 카운트 → `populationBonus` 계산 → Redis/캐시에 1회 갱신. (실시간 DB 조회 배제로 부하 방지)
* 인구 성장에 따른 세계관 확장을 UI·알림으로 스토리텔링.

### 3.6. B2B/B2G 여론조사 파이프라인 (비식별 집계)

* **상품 정의:** "개인의 심리"가 아니라 **"세그먼트별 여론 트렌드 집계"**를 판다. (개인은 안 드러나고 흐름만 보임)
* **법적 설계 — 처음부터:**
  * 가입 시 "AI 학습·통계·B2B 리포트 활용" **별도 동의(opt-in)** 분리 (PIPA 목적별 분리 동의).
  * **OLTP(운영)/OLAP(통계) DB를 1단계 스키마부터 분리**, 새벽 ETL 이관.
  * 외부 반출 전 **집계(aggregate) 단위로만** 가공 + 개인정보성 텍스트 마스킹(정규식/NER).
* **확장형 상품:** 기업/기관이 광고안·정책안을 글로 올리면 → AI 찬반 의견 + 실제 유저 투표로 **여론·반응 테스트**.

## 4. 기술 스택

* **Backend:** FastAPI (비동기)
* **AI 오케스트레이션:** LangChain / LangGraph (진영별 턴 제어, 대댓글 흐름)
* **모델 (OpenAI, 역할별 티어링):**

| 역할 | 모델 | 단가(입/출, 1M토큰) | 비고 |
|---|---|---|---|
| 일반 댓글(대량) | GPT-5.4 Nano | $0.20 / $1.25 | 호출량 압도적 → 단가 핵심 |
| 채점·중재 | GPT-5.4 Mini | $0.75 / $4.50 | 글당 1회, 비중 작음 |
| **캐싱** | 페르소나+원글 prefix 캐싱 | 입력 75~90% 절감 | 댓글 대량 생성 시 필수 |

* **DB:** PostgreSQL(운영) + Redis(가중치·실시간 리밋 캐시) + Vector DB(Chroma 등, 성향 벡터)
* **Async:** FastAPI BackgroundTasks 또는 Celery (딜레이·순차 생성)

## 5. 비용 구조 (검증 완료)

* 댓글 1개 ≈ **0.4원** / 한산한 글(5댓글) ≈ 10원 / 인기 글(200댓글) ≈ 90원.
* 하루 1,000글·글당 평균 21댓글 가정 → **월 약 49만원** 수준.
* 비용 좌우 요인(중요 순): ① 댓글 총량 ② 댓글 1개 출력 길이 ③ 캐싱 적용 여부.
* 계산기: `tools/cost_estimator.py` (가정값 바꿔 재계산 가능).

## 6. BM 로드맵

* **1단계 (B2C):** 몰입형 UX로 유저·리텐션 확보 (토론 연장권 등 인앱 결제 검증).
* **2단계 (B2B 데이터):** 세그먼트별 '주간/월간 여론 트렌드 리포트' 구독.
* **3단계 (B2G/SaaS):** 광고안·정책안 투입 → AI 찬반 + 실제 유저 투표 기반 **가상 여론조사·반응 테스트 베드**.
* **장기 (비용·데이터 해자):** 쌓인 "글 → AI 댓글 → 좋아요/싫어요" 데이터는 그 자체가 학습 교재(좋아요=정답, 싫어요=오답, RLHF 보상 신호와 동형). 충분히 쌓이면 이 데이터로 작은 모델을 학습/증류해 **일반 댓글 생성을 로컬 모델로 전환** → API 비용 최소화 + 외부 의존 탈피 + 모방 불가능한 독점 데이터 해자. 단, 채점·중재 등 고난도 역할은 외부 모델 유지, **일반 댓글부터 단계적 전환.**

## 7. 개발 단계

| 단계 | 프로세스 | 난이도 | 기간 | 핵심 목표 |
| --- | --- | --- | --- | --- |
| 1 | AI 진영 토론 루프 검증 (PoC) | 🟢 하 | 3~5일 | 찬반 AI 토론이 읽을 만하고 객관적인가 |
| 2 | 가변 댓글 엔진 + 백엔드 (MVP) | 🟡 중 | 2~3주 | FastAPI+DB, 채점·딜레이 연출, 상한 없는 리밋 |
| 3 | 인구정보·투표 라벨링·페르소나 진화 | 🔴 상 | 2~3주 | 세그먼트별 투표 데이터 수집, 동적 페르소나 |
| 4 | 배치 + 비식별 집계 파이프라인 | 🟡 중 | 1~2주 | 일일 가중치, OLTP/OLAP 분리, 마스킹 |

### 1단계 — PoC
* OpenAI API 연동. 호위대/도전자/중재자 3~4개 페르소나 프롬프트 작성.
* LangChain으로 [아군 → 적군 반박 → 중재] 고정 턴 체인 구현.
* Streamlit로 화면 최소화, 프롬프트 튜닝에 집중. **실제 토큰 사용량을 측정해 비용 가정 보정.**

### 2단계 — MVP
* FastAPI + PostgreSQL 게시판 API.
* 채점관 로직(0~100) → `BaseLimit` 저장.
* BackgroundTasks로 "등록 완료" 즉시 반환 후 5~10초 딜레이로 댓글 순차 Insert.
* 댓글마다 좋아요/싫어요 재계산 → `FinalLimit` 도달 시 루프 종료.

### 3단계 — 인구·투표·페르소나
* 가입 시 나이대/성별/지역 수집 + 활용 동의(opt-in) 분리.
* 좋아요/싫어요 로그에 [진영·페르소나 태그 + 인구 세그먼트] 동시 기록.
* (초기) 좋아요 태그 단순 합산으로 성향 힌트 → (이후) 임베딩 기반 동적 프롬프트.

### 4단계 — 배치·데이터
* APScheduler로 새벽 4시 MAU 카운트 → `populationBonus` 캐시 갱신.
* 채점 시 감정·산업 카테고리 메타데이터 라벨링.
* OLTP→OLAP ETL + 비식별(마스킹/집계) 전처리.

## 8. 핵심 설계 원칙 (요약)

1. **객관성 우선** — 내 편이 이기게 연출하지 않는다. 찬반 균형 + 사람들의 투표가 판정한다.
2. **진짜 게시판** — 댓글 고정 상한 없음. 점수·반응·인구에 따라 자연스럽게 몇 개~수백 개.
3. **비용은 설계로 통제** — Nano 모델 + 캐싱 + 순차 생성. 검증 결과 상한 없이도 감당 가능.
4. **데이터는 집단으로** — 개인 심리가 아니라 비식별 세그먼트 여론을 판다. 동의·DB분리는 1단계부터.
