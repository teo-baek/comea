# 요구사항: 댓글 단위 반응 (진화 신호 수집)

> **다음 단계 안내**: PRD입니다. 다음 `tech-design`. **북극성** §3 풀의 선행 — 진화 엔진의 입력 신호를 수집한다.

## 1. 배경/목적

목표는 "유저의 **댓글 좋아요/싫어요** 행동으로 AI 페르소나가 진화"(Phase 3)다. 그런데 현재 반응은 **글(스레드) 단위**라(elastic limit 구동), "유저가 *어떤 성향의 댓글*을 선호했는지"를 알 수 없다 — 진화의 핵심 신호가 없다.

이 슬라이스는 그 **신호를 수집**한다: 각 댓글에 좋아요/싫어요를 달 수 있게 하고(유저별), 어떤 페르소나(댓글의 `name`)의 댓글을 선호하는지 데이터로 남긴다. 진화 엔진(다음 슬라이스)이 이 데이터를 +1/-1 합산해 페르소나를 갱신한다. **이번 단계는 수집까지**(진화 미적용).

글 단위 반응(토론 한계선)은 **별도로 그대로 유지**한다.

## 2. 사용자 스토리 / 시나리오

- **독자로서**, 마음에 드는(또는 안 드는) AI 댓글에 좋아요/싫어요를 눌러 표현하고 싶다. 그 선호가 (앞으로) 내 AI를 나에게 맞춰 가는 토대가 된다.

## 3. 기능 요구사항 (FR)

- **FR-1**: `comment_reactions` 테이블 — `(user_id, comment_id, reaction_type, created_at)`. **유저당 댓글당 1개**(재반응 시 갱신/교체).
- **FR-2**: `POST /api/comments/{comment_id}/reaction` (로그인 필수) — `like`/`dislike`. 같은 유저가 같은 댓글에 다시 누르면 **최신 값으로 갱신**(토글/교체). 없는 댓글 404, 잘못된 reaction 400, 무토큰 401.
- **FR-3**: 댓글 반응 **개수는 비노출**(응답/화면에 카운트 없음) — 글쓴이·작성자 보호 원칙 유지.
- **FR-4**: 글 단위 반응(`reactions`, elastic limit 구동)은 **변경하지 않는다**. 댓글 반응은 별도 테이블·별도 경로.
- **FR-5**: Flutter 상세 화면의 **각 댓글 타일에 좋아요/싫어요 버튼**을 추가해 댓글 반응 API를 호출한다. 누른 상태를 시각 피드백(강조)한다.
- **FR-6**: 수집한 데이터(어떤 유저가 어떤 댓글=어떤 페르소나를 선호)는 **진화 엔진(다음 슬라이스)의 입력**이다. 이번엔 수집만, 진화 미적용.

## 4. 비기능 요구사항 (NFR)

한 줄: 댓글 반응은 토론 한계선/댓글 생성에 영향 없음(수집 전용). 속도 비요구.

## 5. 범위 밖 (Out of Scope)

- 진화 엔진(트레잇 갱신) — 다음 슬라이스 (decided: 단순 카운터 합산 + 일일 배치)
- 댓글 반응 개수 노출
- 댓글 반응이 토론 한계선/댓글 생성에 영향
- 대댓글(comment에 다는 comment)
- 반응 취소를 위한 별도 토글 UI(같은 버튼 재탭=교체로 충분)

## 6. 수용 기준 (Acceptance Criteria)

- **AC-1**: 로그인 유저가 특정 댓글에 like/dislike를 보내면 `comment_reactions`에 1건 기록된다.
- **AC-2**: 같은 유저가 같은 댓글에 다시 보내면 새 행이 아니라 **기존 행이 갱신**된다(유저·댓글당 1개 유지).
- **AC-3**: 무토큰 호출은 401, 없는 댓글은 404, 잘못된 reaction은 400.
- **AC-4**: 응답/화면 어디에도 댓글 반응 **개수가 없다.**
- **AC-5**: 글 단위 반응·토론 한계선·댓글 생성 로직은 변하지 않는다(회귀 없음).
- **AC-6**: Flutter 각 댓글에 좋아요/싫어요 버튼이 보이고, 탭하면 API가 호출된다.
