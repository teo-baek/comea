# 요구사항(PRD): 가변적 한계선 + 실시간 반응 (Elastic Comment Limit)

> ⚠️ planned.md(전체 제품 PRD)에서 이 슬라이스를 잘라 합성 + 사용자 피드백으로 개정한 파생 문서.

## 0. 서비스 핵심 (North Star)

**이 서비스의 본질은 "글쓴이가 관심받는다고 느끼게 하는 것"이다.** 글쓴이는 *관심받고자 하는 사람*으로 간주되며, **어떤 글도 소외되어선 안 된다.** 잡담이든 진지한 고민이든 모든 글은 AI 시민들의 댓글을 받는다. 한계선 로직은 "관심의 양"을 조절하는 장치이지, 관심을 차단하는 장치가 아니다.

## 1. 배경 / 문제

현재 `comea_backend`는 글 점수에 따라 **고정된** 댓글 수(4/2/2)를 생성하고, **60점 미만은 0개**(무관심)다. 이는 서비스 핵심(소외 금지)에 정면으로 위배된다. 또한 PRD §3.3의 가변적 한계선·유저 수 기반 확장이 빠져 있다.

## 2. 목표 (이 슬라이스 범위)

모든 글이 유형별 최소 10개 이상의 댓글을 받고, 유저들이 반응(좋아요/싫어요)할수록 토론이 커지며, 전체 유저 규모가 커지면 상한도 함께 커진다. 상한에 도달하면 토론은 조용히 종료된다.

## 3. 기능 요구사항 (FR)

- **FR-1 (전원 댓글 / 소외 금지):** 점수와 무관하게 모든 글은 댓글을 받는다. 유형별 기본 한계선(Base Limit)은 **모두 10 이상**: 잡담(score<60) → **10**, 일반(60≤score<90) → **15**, 명글(score≥90) → **20**.
- **FR-2 (반응 = 관심의 총량):** 좋아요/싫어요 **구분 없이** 반응이 많을수록 토론이 커진다. 증감률은 **총 반응 수**(좋아요+싫어요) 기반. `증감률 = 총반응수 × 0.1`.
- **FR-3 (반응 수치 비노출):** 좋아요/싫어요 개수는 **누구에게도 보이지 않는다**(글쓴이가 상처받지 않도록). API 응답에 반응 카운트를 포함하지 않는다. (B2B 분석용으로 내부 저장은 함.)
- **FR-4 (Final Limit 수식):** `Final Limit = round( Base × (1 + 증감률) )`, `[Base, Effective Cap]` 으로 클램프(최소 Base 보장, 최대 Cap).
- **FR-5 (유저 수 기반 상한):** `Effective Cap = max(고정상한 25, 전체_유저수)`. 유저가 적을 땐 25로 고정, 전체 유저수가 25를 넘어 늘어나면 상한이 유저수에 비례해 확장된다. "전체 유저수"는 설정/캐시 값(`current_population`)에서 읽으며, 실제 일일 집계(새벽 배치)는 **별도 슬라이스**(PRD §3.4)에서 주입한다.
- **FR-6 (실시간 성장):** 반응이 등록되면 Final Limit을 재계산하고, 현재 댓글 수보다 크면 부족분만큼 AI 댓글을 추가 생성한다.
- **FR-7 (조용한 종료, 중재자 없음):** 현재 댓글 수가 Effective Cap에 도달하면 스레드를 잠가(`is_locked = true`) 더 이상 생성하지 않는다. **중재자 AI 요약 댓글은 없다** — 그냥 멈춘다.
- **FR-8 (락 멱등):** 잠긴 스레드는 반응을 계속 기록하되 새 댓글을 생성하지 않는다.
- **FR-9 (동시 클릭 안전 — 스택):** 반응은 카운터 증가가 아니라 **타임스탬프를 가진 개별 레코드(스택)** 로 적재한다. 총 반응 수는 집계(COUNT)로 구한다 → read-modify-write 경합 자체를 제거.
- **FR-10 (속도 비요구):** 일반 게시판처럼 댓글이 즉시 달릴 필요 없다. 생성 지연/동기 호출은 문제로 보지 않는다(성능 최적화 비요구).
- **FR-11 (초기 생성 일관):** 글 등록 시점에도 동일 수식 사용(반응 0 → Final = Base). 기존 하드코딩 4/2/2 제거.
- **FR-12 (페르소나 풀 순환):** 상한이 커도(유저 폭증 시 Cap이 매우 클 수 있음) 댓글을 채울 수 있도록 페르소나 풀(16종)을 순환 재사용한다.
- **FR-13 (댓글 분량 다양화):** 각 AI 댓글의 **분량(길이)을 랜덤하게** 변주한다. 한 줄짜리 짧은 댓글부터 5~6줄의 긴 댓글까지 섞여 실제 게시판처럼 다양하게 보이도록 한다. (분량은 댓글 *개수*와 무관 — 개수는 한계선이, 길이는 변주가 결정.)

## 4. 비범위 (Out of Scope)

- 일일 새벽 배치로 `current_population`을 실제 집계하는 로직 (PRD §3.4, 별도 슬라이스). 이번엔 설정값 주입 훅까지만.
- 유저 인증/가입/식별, 좋아요 1인 1회 제한 (익명 반응 스택).
- 댓글 순차 등장 딜레이 연출 (별도 슬라이스).
- Flutter 클라이언트 반응 버튼 UI (백엔드 API까지만).
- B2B 라벨링/비식별화 (PRD §3.5).

## 5. 수용 기준 (Acceptance)

- AC-1: score 40(잡담) → 댓글 10개, score 70 → 15개, score 95 → 20개. **0개인 글이 없다.**
- AC-2: 반응(좋아요/싫어요 무관)이 쌓이면 Final Limit이 증가해 댓글이 추가 생성된다.
- AC-3: API 응답 어디에도 좋아요/싫어요 **개수가 노출되지 않는다.**
- AC-4: `current_population`이 작을 때(예: 0) 상한은 25. `current_population`을 100으로 올리면 상한이 100으로 확장되어 그 이상 댓글이 가능해진다.
- AC-5: 댓글 수가 Effective Cap에 도달하면 `is_locked=true`가 되고 **중재자 댓글 없이** 종료, 이후 반응해도 댓글 수 불변.
- AC-6: 동시에 여러 반응이 들어와도 각 반응이 개별 레코드로 적재되어 총량 집계가 정확하다(카운터 경합 없음).
- AC-7: 순수 수식 함수는 외부 의존 없이 단위 테스트되고, 모든 AI 호출은 테스트에서 모킹된다.
- AC-8: 댓글 생성 시 길이 스타일이 분량 후보군 중에서 매번 선택되어 프롬프트에 주입된다(짧게/보통/길게). 길이 후보 선택 함수는 시드로 결정적 단위 테스트가 가능하다.
