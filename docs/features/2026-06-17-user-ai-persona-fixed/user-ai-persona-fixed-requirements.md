# 요구사항: 유저별 AI 페르소나 (중간 — 고정, 내부)

> **다음 단계 안내**: 이 문서는 PRD입니다. 다음 단계로 `tech-design`을 호출하세요.
> **북극성 정합:** `docs/architecture/user-ai-persona-north-star.md` §3 "중간(+고정 페르소나)" 슬라이스.

## 1. 배경/목적

북극성의 최종 목표는 "유저 1명당 1개의 고유 AI 에이전트가 **유저의 행동(게시글 작성 + 댓글 좋아요/싫어요)을 기반으로 진화**"하는 것이다(planned.md Phase 3). 그 첫 토대로, **가입 시 유저별 AI 페르소나 레코드를 1개 생성**해 둔다. 이번 단계는 **고정 성격**이며, **사용자에게 노출하지 않는 내부 데이터**다(사용자는 자기 AI의 존재를 몰라도 됨).

진화에 필요한 **행동 데이터는 이미 축적되고 있다**: 유저 인증 슬라이스가 `posts.author_user_id`(누가 썼나)와 `reactions.user_id`(누가 좋아요/싫어요 했나)를 기록한다. 다음(풀) 슬라이스의 진화 엔진이 이 행동 + 본 레코드를 읽어 `trait_params`/프롬프트를 갱신한다. 즉 이 슬라이스는 진화의 **대상 레코드**를 마련하는 단계다.

## 2. 사용자 스토리 / 시나리오

해당 없음 — 내부 데이터 생성이라 외부 사용자 스토리 없음. (사용자는 인지하지 못함.)

## 3. 기능 요구사항 (FR)

- **FR-1**: `ai_personas` 테이블을 둔다 — 유저당 1개(1:1), 표시명/페르소나 프롬프트/성향 파라미터(JSON, 향후 진화용)/갱신시각.
- **FR-2**: **가입 시** 기존 페르소나 풀(`personas.PERSONA_POOL`)에서 **랜덤 1개**를 골라 그 유저의 `ai_personas` 레코드로 생성(이름+프롬프트 저장).
- **FR-3**: 이 페르소나는 **사용자에게 노출하지 않는다** — UI 없음, 사용자용 조회 API 없음. 내부 저장만.
- **FR-4**: 이번 단계에서는 **댓글 생성에 연동하지 않는다**(저장만). 댓글은 기존 공용 풀 그대로.
- **FR-5**: 성향 파라미터(`trait_params`)는 지금은 비워 둔다(풀 단계 진화 엔진이 채움). 레코드 구조만 미리 마련(북극성 §3).

## 4. 비기능 요구사항 (NFR)

한 줄: 가입 트랜잭션에 페르소나 생성을 포함하되, 실패 시 가입까지 깨지지 않도록 안전 처리. 랜덤은 시드 주입 가능(테스트 결정성).

## 5. 범위 밖 (Out of Scope)

- 사용자에게 페르소나 노출(UI/조회 API)
- 댓글 생성에 유저 페르소나 사용("내 AI 출동") — 다음 슬라이스
- 행동 추적 기반 **동적 진화** (풀 단계)
- 가입 설문 기반 성격 결정 (이번엔 풀 랜덤)
- 기존(이미 가입한) 유저 소급 페르소나 생성

## 6. 수용 기준 (Acceptance Criteria)

- **AC-1**: 가입하면 해당 유저의 `ai_personas` 레코드가 1개 생성된다(user_id 연결).
- **AC-2**: 생성된 페르소나의 표시명/프롬프트가 `PERSONA_POOL`의 한 항목과 일치한다.
- **AC-3**: 사용자용 페르소나 조회 엔드포인트가 없다(노출 안 함).
- **AC-4**: 댓글 생성 로직은 변경되지 않는다(기존 공용 풀 유지, 회귀 없음).
- **AC-5**: 한 유저당 페르소나는 1개다(1:1).
